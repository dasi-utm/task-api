# TaskManager.API

Backend API service for a distributed task management system. Handles task CRUD, user authentication, role-based authorization, and publishes domain events to RabbitMQ for consumption by other services.

## Tech Stack

- **.NET 10** — latest SDK features enabled
- **PostgreSQL** — primary data store
- **Entity Framework Core** — Code-First with migrations
- **JWT + bcrypt** — authentication and password hashing
- **RabbitMQ** — domain event publishing (topic exchange)
- **FluentValidation** — request validation
- **Swagger / OpenAPI** — API documentation
- **xUnit** — testing framework

## Architecture — Clean Architecture

Four projects enforcing strict dependency rules. Dependencies flow inward only: API -> Application -> Domain, Infrastructure -> Application -> Domain.

### TaskManager.Domain (innermost — zero dependencies)

Entities, value objects, enums, domain events, and repository interfaces.

- **Entities** live in `Entities/`. Each entity inherits from `BaseEntity` which provides `Id`, `CreatedAt`, `UpdatedAt`, `CreatedBy`, `IsDeleted`.
- **Value objects** live in `ValueObjects/`. Immutable, compared by value.
- **Domain events** live in `Events/`. Named `{Entity}{Action}Event` (e.g., `TaskCreatedEvent`). Raised inside entity methods, never from outside.
- **Enums** live in `Enums/`.
- **Repository interfaces** live in `Interfaces/`. Define data access contracts only — no EF Core types here.

Rules:
- NO `using` statements referencing Application, Infrastructure, or API.
- NO dependency injection, no framework types.
- Business rules are enforced inside entity methods.
- Entities raise domain events via a `DomainEvents` collection on `BaseEntity`.

### TaskManager.Application (use cases)

Commands, queries, handlers, DTOs, validators, domain event handlers, and infrastructure service interfaces.

- **Commands** in `Commands/{Entity}/` — named `Create{Entity}Command`, `Update{Entity}Command`, `Delete{Entity}Command`.
- **Queries** in `Queries/{Entity}/` — named `Get{Entity}Query`, `Get{Entities}Query`.
- **Handlers** colocated with their command/query — named `{Command}Handler`, `{Query}Handler`.
- **DTOs** in `DTOs/` — named `{Entity}Dto`, `Create{Entity}Dto`, `Update{Entity}Dto`.
- **Validators** in `Validators/` — FluentValidation, named `{Command}Validator`.
- **Domain event handlers** in `EventHandlers/` — named `{Event}Handler`.
- **Service interfaces** in `Interfaces/` — `IEventPublisher`, `ITokenService`, `ICurrentUserService`.
- **Common** in `Common/` — `Result<T>` pattern, pagination types.

Rules:
- Depends only on Domain.
- No direct DbContext usage. Access data through repository interfaces from Domain.
- Returns `Result<T>` for all operations (no throwing exceptions for business logic).
- Handlers are single-responsibility: one handler per command or query.

### TaskManager.Infrastructure (external concerns)

EF Core DbContext, entity configurations, migrations, repository implementations, JWT service, RabbitMQ publisher.

- **Data** in `Data/` — `AppDbContext`, `EntityTypeConfiguration` classes in `Data/Configurations/`.
- **Migrations** in `Data/Migrations/` — auto-generated by EF Core.
- **Repositories** in `Repositories/` — implement interfaces from Domain.
- **Services** in `Services/` — `TokenService` (JWT), `EventPublisher` (RabbitMQ), `CurrentUserService`.

Rules:
- Depends on Domain and Application.
- Only project that references EF Core and database packages.
- Each entity gets its own `EntityTypeConfiguration` class.
- Soft delete implemented via global query filter on `IsDeleted`.
- Domain events dispatched after `SaveChangesAsync` succeeds (override in DbContext).

### TaskManager.API (outermost — composition root)

Controllers, middleware, dependency injection, Swagger configuration, `Program.cs`.

- **Controllers** in `Controllers/` — thin, max 5-10 lines per action. Named `{Entity}Controller`.
- **Middleware** in `Middleware/` — `ExceptionHandlingMiddleware`, `AuthenticationMiddleware`.
- **Extensions** in `Extensions/` — DI registration extension methods (e.g., `AddInfrastructure`, `AddApplication`).
- `Program.cs` — composition root. Registers all services, configures pipeline.

Rules:
- Controllers only orchestrate: receive request, call handler, return response.
- No business logic in controllers.
- All cross-cutting concerns (auth, error handling, rate limiting) handled in middleware.

## Patterns

### CQRS-Lite

Separate command and query models but a single data store. Commands mutate state and return `Result<T>`. Queries are read-only and return DTOs directly.

### Domain Events

1. Entity method performs business logic and adds event to `DomainEvents` collection.
2. `SaveChangesAsync` override in `AppDbContext` persists changes.
3. After successful commit, dispatches collected domain events to handlers.
4. Event handlers may publish to RabbitMQ via `IEventPublisher`.

### Result Pattern

All business operations return `Result<T>` instead of throwing exceptions.

```csharp
// Success
return Result<TaskDto>.Success(taskDto);

// Failure
return Result<TaskDto>.Failure("Task not found");
```

Controllers translate Result to HTTP responses:
- `Result.IsSuccess` -> 200/201
- `Result.IsFailure` -> 400/404/409 based on error type

### Repository Pattern

- Interfaces defined in Domain (`ITaskRepository`, `IUserRepository`).
- Implementations in Infrastructure using EF Core.
- No `DbContext` usage outside Infrastructure.
- Generic base repository `IRepository<T>` for common CRUD.

### Thin Controllers

Every controller action follows this pattern:

```csharp
[HttpPost]
public async Task<IActionResult> Create([FromBody] CreateTaskDto dto)
{
    var command = new CreateTaskCommand(dto, CurrentUserId);
    var result = await _handler.Handle(command);
    return result.IsSuccess
        ? CreatedAtAction(nameof(GetById), new { id = result.Value.Id }, result.Value)
        : BadRequest(new ErrorResponse(result.Error));
}
```

## API Design

### URL Structure

All endpoints follow: `api/v1/{resource}`

```
GET    api/v1/tasks          — list tasks (paginated)
GET    api/v1/tasks/{id}     — get task by ID
POST   api/v1/tasks          — create task
PUT    api/v1/tasks/{id}     — update task
DELETE api/v1/tasks/{id}     — soft-delete task
PATCH  api/v1/tasks/{id}/status — change task status

POST   api/v1/auth/register  — register user
POST   api/v1/auth/login     — login, returns JWT
```

### Error Response Format

```json
{
  "error": "Descriptive error message",
  "details": {
    "field": ["Validation error 1", "Validation error 2"]
  }
}
```

### Pagination

Request: `?page=1&pageSize=20`

Response:
```json
{
  "data": [],
  "total": 100,
  "page": 1,
  "pageSize": 20
}
```

### Swagger / OpenAPI

- All controller actions have XML doc comments.
- `[ProducesResponseType]` attributes on every action.
- Request/response examples where helpful.

## Database

### EF Core Code-First

- Each entity has a separate `EntityTypeConfiguration` class in `Infrastructure/Data/Configurations/`.
- Migrations generated in `Infrastructure/Data/Migrations/`.
- Connection string in `appsettings.json` under `ConnectionStrings:DefaultConnection`.

### Audit Fields (on every entity via BaseEntity)

| Field | Type | Behavior |
|-------|------|----------|
| `Id` | `Guid` | Auto-generated |
| `CreatedAt` | `DateTime` | Set on creation (UTC) |
| `UpdatedAt` | `DateTime` | Set on every update (UTC) |
| `CreatedBy` | `string` | Set from current user on creation |
| `IsDeleted` | `bool` | Soft delete flag |

### Soft Delete

- Global query filter: `.HasQueryFilter(e => !e.IsDeleted)`
- Delete operations set `IsDeleted = true` and `UpdatedAt = DateTime.UtcNow`.
- Include deleted records only when explicitly needed via `.IgnoreQueryFilters()`.

## Event Publishing (RabbitMQ)

### Exchange and Routing

| Setting | Value |
|---------|-------|
| Exchange | `task-events` |
| Exchange type | `topic` |
| Routing keys | `task.created`, `task.updated`, `task.deleted`, `task.status-changed` |

### Event Message Schema

```json
{
  "eventType": "TaskCreated",
  "timestamp": "2026-03-09T12:00:00Z",
  "correlationId": "guid-here",
  "payload": {
    "taskId": "guid",
    "title": "string",
    "assignedTo": "guid"
  }
}
```

### Flow

1. Entity raises domain event inside business method.
2. `AppDbContext.SaveChangesAsync` commits to PostgreSQL.
3. On success, domain event dispatcher invokes registered handlers.
4. `EventPublisher` (in Infrastructure) serializes event and publishes to RabbitMQ.
5. On DB failure, no event is published (no inconsistency).

## Commands

### Build and Run

```bash
# Build the solution
dotnet build

# Run the API (with hot reload)
dotnet watch run --project TaskManager.API

# Run without hot reload
dotnet run --project TaskManager.API
```

### Testing

```bash
# Run all tests
dotnet test

# Run with coverage
dotnet test --collect:"XPlat Code Coverage"

# Run specific test project
dotnet test TaskManager.Tests/TaskManager.Tests.csproj

# Run tests matching a filter
dotnet test --filter "FullyQualifiedName~CreateTask"
```

### Database Migrations

```bash
# Add a new migration
dotnet ef migrations add MigrationName --project TaskManager.Infrastructure --startup-project TaskManager.API

# Apply migrations to database
dotnet ef database update --project TaskManager.Infrastructure --startup-project TaskManager.API

# Remove last migration (if not applied)
dotnet ef migrations remove --project TaskManager.Infrastructure --startup-project TaskManager.API

# Generate SQL script (for review)
dotnet ef migrations script --project TaskManager.Infrastructure --startup-project TaskManager.API
```

### Docker

```bash
# Start all services (API, PostgreSQL, RabbitMQ)
docker-compose up -d

# Rebuild and start
docker-compose up -d --build

# View logs
docker-compose logs -f api

# Stop all services
docker-compose down

# Stop and remove volumes (clean DB)
docker-compose down -v
```
